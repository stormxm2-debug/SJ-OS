# SJ OS — 공지사항 가이드

회사 공지 시스템입니다. 대표/관리자가 공지를 작성/게시하고, 직원은 자신에게 해당하는 게시된
공지만 봅니다. 최종 접근 제어는 RLS가 적용합니다.

## 구조

- `announcements`: 공지 본문 + 대상 + 상태 + 고정.
- `announcement_reads`: 직원별 읽음 기록 (본인 것만 기록 가능).

## 대상(타겟)

- **전체 공지** (`all`): 모든 활성 직원.
- **역할별 공지** (`role`): `target_role`과 프로필 역할이 일치하는 직원.
- **팀별 공지** (`team`): `target_team_id`와 프로필 팀이 일치하는 직원.

## 중요도 / 고정

- 중요도: 일반 / 중요 / 긴급 (긴급은 목록에서 강조).
- 상단 고정: `pinned=true` 공지는 목록 최상단.
- 정렬: 고정 → 중요도(긴급>중요>일반) → 게시일 최신순.

## 상태

- **임시저장**(draft): 작성자만(=관리자) 목록에서 보임, 직원에게 안 보임.
- **게시됨**(published): 대상 직원에게 노출.
- **숨김**(hidden) / **보관**(archived): 직원에게 안 보임.

## 읽음 처리

- 직원이 공지를 열면 `announcement_reads`에 본인 기록이 추가됩니다(중복 방지 upsert).
- 읽지 않은 공지 수가 홈/공지 화면에 표시됩니다.
- 다른 사용자의 읽음 기록은 만들 수 없습니다(`profile_id = auth.uid()`).

## 권한

- 대표/관리자: 작성/게시/숨김/보관/삭제 + 전체 조회.
- 팀장/FC: 본인 대상 게시 공지 조회 + 본인 읽음 처리만. **공지 관리 불가**
  (공지사항 관리 라우트 접근 시 접근 거부).

## RLS 주의사항

- 직원은 게시(published)되고 자신에게 타겟된 공지만 조회. draft/hidden/archived는 노출 금지.
- 공지 작성/수정/삭제는 owner/admin만. 공개(public) 조회 정책 금지. RLS 비활성화 금지.
- 렌더러는 service_role을 사용하지 않습니다.

## Supabase SQL 적용 순서

1. 기본 스키마(profiles/teams) + `SJ_OS_SUPABASE_SCHEMA.sql`.
2. `SJ_OS_SUPABASE_ANNOUNCEMENTS_SCHEMA.sql` (테이블/인덱스).
3. `SJ_OS_SUPABASE_ANNOUNCEMENTS_RLS.sql` (RLS 활성화 + 정책).

## E2E 테스트 (로그인 후)

1. 관리자 로그인 → 공지사항 관리 → 전체/역할별/팀별 공지 게시.
2. FC 로그인 → 공지사항에서 대상 공지만 보이는지 확인, 열면 읽음 처리.
3. FC가 공지사항 관리 라우트 접근 시 "접근 권한이 없습니다." 확인.
