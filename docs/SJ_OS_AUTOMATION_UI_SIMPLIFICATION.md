# SJ OS 자비스 자동개발 화면 단순화

대표/관리자가 헷갈리지 않도록 **자비스 자동개발(개발 프롬프트 센터)** 화면을
단순화했습니다. 자동화 엔진의 동작 로직(파일 생성 · typecheck · build · git status)은
그대로 두고, **UI 정리만** 수행했습니다.

---

## 1. 기존 화면이 왜 헷갈렸는가

한 화면(`DeveloperPromptCenterPage`)에 서로 목적이 겹치는 패널이 10개 가까이 세로로
쌓여 있었습니다.

- `Claude 자동 개발`(명령→실행), `Claude Code 실행 승인`(승인→명령→실행) 두 곳 모두
  **“Claude Code 실행”** 버튼이 있어, 같은 이름의 버튼이 서로 다른 동작을 했습니다.
- 실행 버튼 이름이 `Claude Code 실행` · `지금 실행` · `다시 시도` · `다음 작업 실행`
  으로 제각각이었습니다.
- 승인 버튼이 **“승인 후 실행 명령 생성”** 이라, 실제로 실행되는지 헷갈렸습니다.
- `큐 자동 실행 ON/OFF`, `큐 일시정지/재개`, `다음 작업 실행` 등 큐 제어 버튼이 항상
  노출되어 화면이 복잡했습니다.
- `Claude Code CLI 확인`, `실행 환경 확인`, `실행 환경 없음` 등 진단/환경 관련
  요소가 여러 패널에 흩어져 있었습니다.
- 자동 복구 · 커밋/푸시 · 완료 리포트 · 병렬/워크트리 · 개발 프롬프트 칸반까지 한 번에
  보여, 처음 쓰는 대표/관리자는 어디서 시작할지 알기 어려웠습니다.

## 2. 새 사용 순서 (일반 화면)

일반 화면에는 딱 이 흐름만 보입니다.

1. **명령 입력** — 예: “직원 출퇴근 기능 만들어줘” → `작업 생성`
2. **프롬프트 미리보기** — 실행될 Claude Code 프롬프트를 그대로 확인
3. **승인하고 실행** — 누르면 Electron Main이 실제로 Claude Code를 실행
4. **실행 로그** — 아래 고정 순서로 진행 상황 표시
5. **결과 확인** — typecheck / build / exit code / `git status --short` 결과 확인

### 일반 화면에 남긴 버튼(딱 7개)

- `작업 생성`
- `프롬프트 미리보기`
- `승인하고 실행`
- `실행 로그 보기`
- `중지`
- `다시 실행`
- `고급 설정 열기`

### 실행 로그 표시 순서 (고정)

실행 로그를 열면 아래 순서의 단계 가이드가 표시됩니다. 이 순서는 러너(Electron Main)가
실제로 출력하는 로그 순서와 동일합니다.

1. 작업 생성됨
2. 승인됨
3. Claude Code 실행 시작
4. cwd 확인
5. 프롬프트 전달
6. Claude Code 완료 / exit code
7. typecheck 시작 / 결과
8. build 시작 / 결과
9. git status 결과
10. 완료 / 실패

## 3. 대표/관리자 사용법

1. 좌측 메뉴 **관리자 · 개발 → 개발 프롬프트 센터(자비스 자동개발)** 로 이동합니다.
2. 상단 **사용 순서** 카드에서 5단계 흐름을 확인합니다.
3. 명령을 입력하고 `작업 생성` 을 누릅니다. (실행 환경은 자동으로 점검됩니다.)
4. 작업 카드에서 `프롬프트 미리보기` 로 내용을 확인합니다.
5. `승인하고 실행` 을 누르면 실제 실행이 시작됩니다.
6. `실행 로그 보기` 로 단계별 진행과 검증 결과를 확인합니다.
7. 결과(typecheck/build/git status)를 확인하고, 커밋/푸시가 필요하면 아래 “왜 터미널에서
   직접 하는지”를 참고하세요.

## 4. FC / 팀장 접근 차단

- 자비스 자동개발은 **대표/관리자 전용**입니다.
- 라우트 가드(`canAccessRoute`)에서 이미 admin 전용으로 분류되어 FC/팀장 메뉴에는
  노출되지 않으며, 화면에서도 역할을 다시 확인합니다(`isAdminRole`).
- FC/팀장이 직접 접근하면 **“자비스 자동개발은 대표/관리자 전용입니다.”** 안내 카드가
  표시되고 기능은 차단됩니다.

## 5. Web / PWA 차단

- 실제 파일 생성/실행은 **데스크톱(Electron) 앱에서만** 가능합니다.
- 렌더러의 `isElectronRuntime()` + 모바일 뷰포트 판별로 데스크톱 여부를 확인합니다.
- Web/PWA(모바일)에서 접근하면 **“자비스 자동개발은 PC앱 전용입니다.”** 안내가 표시됩니다.

## 6. 고급 설정에 들어간 기능

`고급 설정 열기` 를 눌러야 보이며, 일반 사용에는 필요하지 않습니다.

- **실행 환경 진단** (`Claude Code 실행 환경`) — 실행 환경 확인 · 실행 테스트
- **실행 승인 러너** (`Claude Code 실행 승인`) — 개발 프롬프트 패킷 승인 → 명령 생성
- **러너 스모크 테스트** — Git 상태/커밋, Typecheck, Build, Web Build, **Claude Code CLI 확인**
- **Claude 자동개발 큐** — 큐 자동 실행 ON/OFF · 일시정지/재개 · 다음 작업 실행 · 큐에서 취소
- **자동 복구** — 실패한 검증에 대한 복구 작업(승인 필요)
- **커밋 / 푸시** — 성공한 작업의 승인형 커밋/푸시(2단계)
- **완료 리포트** — 릴리스 노트 · 수동 테스트 체크리스트
- **병렬 / 워크트리** — worktree 기반 병렬 빌더 및 결과 검토(읽기 전용)
- **개발 프롬프트 센터(칸반)** — 프롬프트 큐 상태 추적 · 내보내기 · 초기화

## 7. 커밋 / 푸시는 왜 터미널에서 직접 하는지

- 자동화 엔진(러너)은 **커밋/푸시를 자동으로 실행하지 않습니다.** 파일 생성과
  typecheck / build / `git status --short` 검증까지만 수행합니다.
- 커밋/푸시는 결과를 사람이 눈으로 확인한 뒤 진행해야 하는 **되돌리기 어려운 작업**이며,
  특히 `main` 브랜치 직접 푸시나 강제 푸시는 안전 가드로 막혀 있습니다.
- 따라서 검증 통과 후에도 **대표/관리자가 터미널에서 직접** 아래처럼 실행합니다.

```bash
git status            # 변경 확인
git add <파일>        # 검토한 파일만 스테이징
git commit -m "..."   # 의미 있는 메시지로 커밋
git push origin <branch>   # main 직접 푸시 대신 브랜치 푸시 권장
```

> 참고: 고급 설정의 “커밋 / 푸시” 패널은 검증을 통과한 작업에 한해 **승인형 2단계**로
> 보조하지만, 최종 판단과 실행 책임은 사람에게 있습니다. `.env`, `.env.local` 등 민감
> 파일은 어떤 경우에도 자동으로 건드리지 않습니다.
